ТЕХНИЧЕСКОЕ ЗАДАНИЕ
Разработка веб-приложения "Семейный финансовый планировщик"
Версия документа: 3.0 (Финальная)
Дата составления: 2024 год
1. Введение
1.1. Наименование проекта
Семейный финансовый планировщик "Резерв"
1.2. Цель проекта
Разработка веб-приложения для долгосрочного планирования семейного бюджета, позволяющего избежать кассовых разрывов за счет распределения периодических расходов, контроля накоплений и анализа финансовой истории.
1.3. Описание проекта (Идея)
Приложение позволяет пользователям планировать доходы и расходы на год вперед, с особым акцентом на управление периодическими платежами и накоплениями. Ключевой особенностью является механизм "подтверждения" месяца, который фиксирует фактические операции и отделяет план от реальности, обеспечивая точный исторический анализ.
1.4. Целевая аудитория
Частное некоммерческое приложение для ограниченного круга лиц.
1.5. Глоссарий
● ОЕП - Обязательные Ежемесячные Платежи
● ПП - Плановые Платежи (периодические, не ежемесячные)
● Свободный остаток - Деньги, доступные для траты после всех обязательств
● Виртуальный счет - Показатель, соответствующий реальной сумме на банковском счете
● Зарезервированные средства - Сумма, которую пользователь обязался не тратить
● Подтверждение операции - Фактическое совершение запланированной операции в реальности
● Закрытие месяца - Фиксация всех подтвержденных операций за месяц
● Дашборд - Основная страница приложения с сводной информацией
2. Требования к функциональности
2.1. Модуль управления доходами
2.1.1. Типы доходов:
Основной ежемесячный доход:
● Настройка в профиле пользователя
● Учитывается в бюджете текущего и всех последующих месяцев
● Типы основных доходов:
○ Бессрочный: пользователь указывает размер и наименование дохода
○ Временный: пользователь указывает размер и период (две даты, начало и конец)
● Валидация: Только положительные числа
Дополнительный (разовый) доход:
● Добавляется через форму на дашборде для текущего месяца
● Влияет только на бюджет текущего месяца
● Валидация: Только положительные числа
2.2. Модуль управления расходами
2.2.1. Типы расходов:
ОЕП (Обязательные ежемесячные платежи):
● Пользователь создает статью расхода с указанием названия и суммы
● Типы ОЕП:
○ Бессрочный: пользователь указывает размер и наименование
○ Временный: пользователь указывает размер и период (две даты, начало и конец)
● При подтверждении: сумма списывается с Виртуального счета
● Валидация: Только положительные числа
ПП (Плановые платежи):
● Пользователь создает статью с указанием:
○ Наименование
○ Сумма
○ Дата целевого платежа
○ Периодичность (в месяцах)
● Система рассчитывает ежемесячный платеж: Сумма / Периодичность
● При подтверждении: система фиксирует резервирование средств (без изменения ВС)
● При достижении цели: сумма списывается с Виртуального счета
● Валидация: Только положительные числа
Накопления:
Режим 1: "Как можно скорее"
● Пользователь указывает целевую сумму
● Система фиксирует ежемесячный платеж, равный текущему свободному остатку
● Расчет количества месяцев: Целевая сумма / Свободный остаток (округление вверх)
● Платеж фиксируется на весь срок
● При подтверждении: система фиксирует резервирование средств (без изменения ВС)
● При достижении цели: сумма списывается с Виртуального счета
Режим 2: "К определенной дате"
● Пользователь указывает целевую сумму и дату
● Алгоритм: Ежемесячный платеж = Целевая сумма / Кол-во месяцев до даты
● При подтверждении: система фиксирует резервирование средств (без изменения ВС)
● При достижении цели: сумма списывается с Виртуального счета
● Валидация: Платеж не должен превышать свободный остаток
Режим 3: "Без цели"
● Пользователь указывает фиксированную ежемесячную сумму
● При подтверждении: система фиксирует резервирование средств (без изменения ВС)
● Валидация: Сумма не должна превышать свободный остаток
2.3. Модуль управления бюджетом (Дашборд)
2.3.1. Отображение данных:
● Бюджет на текущий (активный) месяц
● Список плановых доходов и расходов
● Виртуальный счет (соответствует реальному банковскому счету)
● Зарезервированные средства
● Свободный остаток
● Информация о бюджетах на будущие месяцы
● Суммы по типам расходов (ОЕП, ПП, Накопления)
● Для каждой статьи ПП и Накоплений: цель, дата, уже накоплено, осталось
2.3.2. Процесс подтверждения и закрытия месяца:
Подтверждение операций:
Подтвердить:
● ОЕП: сумма списывается с Виртуального счета
● ПП и Накопления: система фиксирует резервирование средств
Изменить и подтвердить: Внести фактическую сумму
● Для ПП и Накоплений:
○ При сумме меньше плановой: автоматический перенос даты окончания цели
○ При сумме больше плановой: выбор стратегии (ускорить цель/уменьшить платеж)
○ При последующей компенсации: автоматический возврат к первоначальному плану при возможности
● Для ОЕП:
○ При сумме меньше плановой: автоматический перенос дефицита на следующий месяц
○ При сумме больше плановой: фиксация суммы в текущем месяце, списание с Виртуального счета
Отменить:
● Платеж не был совершен
● Приравнивается к алгоритму "При сумме меньше плановой" для соответствующего типа расхода
Добавление фактических операций:
● Пользователь может добавить незапланированные доходы или расходы за текущий месяц
Распределение свободного остатка:
● "Списать": Остаток считается потраченным, списывается с Виртуального счета
● "Оставить": Остаток переносится в качестве дополнительного дохода на следующий месяц
Завершение:
● Пользователь нажимает кнопку "Закрыть месяц"
● Данные фиксируются в Истории бюджетов
● Становится доступен следующий месяц
● ВАЖНО: Месяц не может быть закрыт с отрицательным свободным остатком
2.4. Модуль аналитики и истории
● История бюджетов: Снимки закрытых месяцев "как есть", без возможности изменения
● История операций: Логирование всех действий пользователей в системе
2.5. Модуль пользователей и прав
● Регистрация: Пользователей добавляет разработчик
● Создание бюджета: Первый пользователь становится Владельцем
● Приглашение пользователей: Владелец указывает email пользователя для доступа к бюджету
● Права доступа: Все пользователи имеют равные права, кроме удаления бюджета и пользователей (только Владелец)
3. Алгоритмы и расчеты
3.1. Алгоритм подсчета дохода в текущем месяце (План)
Доход (план) = Σ(Основной доход всех активных пользователей) + Σ(Дополнительные доходы текущего месяца)
3.2. Алгоритм подсчета расходов в текущем месяце (План)
● Расходы ОЕП (план) = Σ(Все активные статьи ОЕП)
● Расходы ПП (план) = Σ(Платежи по статьям ПП, у которых в текущем месяце есть платеж)
● Расходы Накопления (план) = Σ(Ежемесячные платежи по всем активным статьям накоплений)
● Общие расходы (план) = Расходы ОЕП + Расходы ПП + Расходы Накопления
3.3. Алгоритм Виртуального счета
text
Виртуальный счет = 
  Начальный баланс (с предыдущего месяца)
  + Σ(Все доходы месяца) 
  - Σ(Все подтвержденные ОЕП) 
  - Σ(Списанные цели ПП и Накоплений)
  - Σ(Списанный свободный остаток)
3.4. Алгоритм зарезервированных средств
text
Зарезервированные средства = 
  Σ(Все подтвержденные, но не списанные ПП и Накопления)
3.5. Алгоритм свободного остатка
text
Свободный остаток = Виртуальный счет - Зарезервированные средства
3.6. Обработка отклонений от плана
Для ПП и Накоплений:
● Недовнесение: Новая_дата = Текущая_дата + (Оставшаяся_сумма / Фиксированный_платеж)
● Переплата: Пользователь выбирает стратегию применения излишка
Для ОЕП:
● Недовнесение: Дефицит переносится на следующий месяц
● Переплата: Сумма фиксируется в текущем месяце
4. Нефункциональные требования
4.1. Юзабилити
Интерфейс должен быть интуитивно понятен для пользователей без финансового образования
4.2. Производительность
Загрузка дашборда и отчетов не должна превышать 3 секунд
4.3. Безопасность
● Обязательное использование HTTPS
● Пароли хранятся в хешированном виде
● Защита финансовых данных пользователей
4.4. Надежность
Обеспечение целостности данных, особенно в процессе закрытия месяца
5. Бизнес-логика и валидация
5.1. Принципы системы
● Виртуальный счет всегда соответствует реальному банковскому счету
● Резервирования не создают "денег из воздуха"
● Свободный остаток показывает реально доступные для траты средства
● Месяц не может быть закрыт с отрицательным остатком
5.2. Валидация данных
● Запрет нулевых и отрицательных значений в денежных полях
● Запрет создания расходов, приводящих к отрицательному остатку
● Контроль соответствия Виртуального счета и реального банковского счета
5.3. Обработка исключительных ситуаций
● Автоматический перенос дат при недовнесении платежей
● Уведомления об изменениях в планах
● Предупреждения о приближающихся платежах
