## 2. Цели и контекст проекта

### 2.1 Проблема, которую решает проект

При накоплении средств на банковских счетах возникает эффект «большой кучи»: пользователь видит общую сумму, но не понимает, сколько из неё уже «занято» под будущие обязательства, а сколько можно свободно тратить. Это приводит к кассовым разрывам — деньги потрачены, а когда подходит срок страховки, налога или другого крупного платежа, средств не хватает.

Приложение «Резерв» решает задачу **«всегда достаточно денег для любого платежа»** — показывает, сколько из накопленной суммы реально свободно, и резервирует нужные части под конкретные цели.

Ключевой принцип: управление **«широкими мазками»** — без детализации до уровня чека. Пользователь планирует и контролирует финансы по категориям, сохраняя финансовую дисциплину без утомительной ежедневной сверки.

### 2.2 Цель проекта

Разработать веб-приложение для долгосрочного планирования семейного бюджета, позволяющее избежать кассовых разрывов за счёт резервирования средств под периодические расходы и накопления, с чётким разделением плана и факта.

### 2.3 Ключевая ценность

- **Механизм Виртуального счёта** — всегда соответствует реальному остатку в банке
- **Резервирование под будущие цели** — деньги «заморожены», но не списаны
- **Подтверждение операций** — план отделён от реальности
- **Закрытие месяца** — фиксирует историю и обеспечивает преемственность периодов
- **Хронология бюджетов** — таблица на 3 месяца назад + текущий + 12 вперёд
- **Контроль параллельной работы** — last-write-wins в v1; разрешение конфликтов в v2+
- **Механизм кредитования из накоплений** — расчёт доступной суммы и срока (v3+)

### 2.4 Границы проекта — что НЕ входит в scope

- Не является бухгалтерским инструментом
- Не синхронизируется с банками автоматически — ввод данных вручную
- Не поддерживает несколько валют (только рубли)
- Не является публичным сервисом — частное приложение для ограниченного круга лиц
- Без push-уведомлений и email-рассылок в первом релизе

## 3. Пользователи системы

### 3.1 Роли пользователей

| Роль | Описание | Права |
|---|---|---|
| Владелец | Создатель бюджета | Все права: просмотр, редактирование, подтверждение операций, закрытие месяца, управление пользователями, удаление бюджета |
| Участник | Приглашённый пользователь | Все права Владельца, кроме удаления бюджета и управления пользователями |

> **Примечание:** Разница в пользовательских сценариях между Владельцем и Участником отсутствует — оба совершают одинаковые действия с бюджетом.

### 3.2 Добавление и управление пользователями

- **v1:** создание учётных записей — разработчик напрямую в БД
- **v2+:** самостоятельная регистрация (signup) через интерфейс
- Владелец удаляет участника через интерфейс
- При удалении аккаунта Владельца: статус Владельца переходит к следующему пользователю по ID
- Удаление последнего пользователя ведёт к удалению всего бюджета
- Явный выбор преемника при передаче роли Владельца — v2+

### 3.3 Ключевые пользовательские сценарии

> **Пример:** Пользователь открывает дашборд и сразу видит: ВС = 85 000 ₽, Зарезервировано = 32 000 ₽, Свободный остаток = 53 000 ₽. Понимает, что может потратить не более 53 000 ₽.

> **Пример:** Пользователь создаёт накопление «На отпуск к августу» — 60 000 ₽. Система рассчитывает ежемесячный взнос, проверяет свободный остаток в каждом месяце до августа и либо создаёт запись, либо показывает ошибку с объяснением.

> **Пример:** Пользователь оплатил ипотеку в реальности, открывает приложение и нажимает «Списать» напротив статьи ипотеки. ВС уменьшается. Строка окрашивается в статус «Списан».

> **Пример:** Пользователь в марте подтвердил все платежи и доходы, нажимает «Закрыть месяц». Март уходит в историю, апрель становится текущим, таблица сдвигается влево, добавляется новый месяц в конце.

> **Пример:** Пользователь начинает работу с приложением. У него уже есть 120 000 ₽ на счету. Он создаёт разовый доход «Текущий капитал» на 120 000 ₽ и подтверждает его — ВС сразу отражает реальное состояние счёта.

## 4. Глоссарий

| Термин | Определение |
|---|---|
| ОЕП | Обязательные Ежемесячные Платежи — регулярные расходы каждый месяц (аренда, ЖКХ, кредиты). Философия: критические статьи расходов, требующие ежемесячного контроля |
| ПП | Плановые Платежи — периодические расходы реже раза в месяц (страховка, налоги, техосмотр). Философия: заморозить средства равными долями, чтобы избежать шоковой нагрузки в месяц платежа |
| Накопление | Целевое резервирование средств для будущей крупной покупки или события. Философия: превратить «гору денег» в упорядоченные цели |
| Виртуальный счёт (ВС) | Расчётный показатель, соответствующий реальному остатку на банковском счёте. Изменяется только при подтверждении операций пользователем |
| Зарезервированные средства | Сумма подтверждённых, но ещё не списанных взносов по ПП и Накоплениям. Деньги физически на счету, но тратить их нельзя |
| Свободный остаток (СО) | Деньги, реально доступные для трат прямо сейчас: ВС − Зарезервированные средства |
| Плановый свободный остаток | Прогнозный СО на основе запланированных (но ещё не подтверждённых) доходов и расходов |
| Подтверждение операции | Фиксация пользователем факта совершения платежа или получения дохода в реальности. Изменяет ВС |
| Списание | Уменьшение Виртуального счёта на сумму подтверждённой операции |
| Закрытие месяца | Финальная фиксация всех операций периода. Месяц переходит в историю (read-only) |
| Активный месяц | Текущий рабочий месяц, открытый для подтверждения операций |
| Статус платежа | Одно из трёх состояний: **Запланирован / Списан / Отменён** |
| Статус дохода | Одно из двух состояний: **Запланирован / Подтверждён** |
| PDD | Prompt-Driven Development — методология разработки, при которой ТЗ используется как контекст для генерации кода через ИИ |

## 5. Функциональные требования

Каждое требование имеет уникальный код `FR-XX.YY`. Все перечисленные требования считаются ожидающими реализации, если явно не указано иное. Требования с пометкой `(v2+)` реализуются в будущих релизах.

### FR-01 · Аутентификация и управление доступом
*Безопасный вход в систему и разграничение прав пользователей*

- **FR-01.1** Вход по email и паролю
- **FR-01.2** Выход из системы
- **FR-01.3** Восстановление пароля через email (v2+)
- **FR-01.4** Создание пользователя разработчиком напрямую в БД (v1)
- **FR-01.5** Приглашение участника Владельцем по email (v2+)
- **FR-01.6** Удаление пользователя из бюджета — только Владелец
- **FR-01.7** Удаление бюджета целиком — только Владелец
- **FR-01.8** Двухфакторная аутентификация (v2+)

> **Примечание:** Сессия активна 12 часов, после чего пользователь должен войти повторно.

### FR-02 · Первоначальная настройка (онбординг)
*Помощь новому пользователю настроить бюджет при первом входе*

Онбординг служит удобству, но не является обязательным шагом для начала работы.

- **FR-02.1** Обучающий экран при первом входе: краткое объяснение ВС, Зарезервированных средств и Свободного остатка (v2+)

> **Примечание:** Начальный баланс вводится через разовый доход: пользователь создаёт статью «Текущий капитал» на нужную сумму и подтверждает её. ВС сразу отражает реальное состояние счёта.

---

### FR-03 · Управление доходами
*Учёт всех источников дохода для корректного расчёта бюджета каждого месяца*

> **Примечание:** Доход требует подтверждения пользователем — означает, что деньги реально поступили на счёт. Только подтверждённые доходы увеличивают ВС.

#### Основной доход

- **FR-03.1** Создание бессрочного основного дохода (название + сумма). По умолчанию активен
  > Пример: «Зарплата» — 120 000 ₽ / месяц, бессрочно
- **FR-03.2** Создание временного основного дохода (название, сумма, дата начала, дата окончания)
  > Пример: «Проектный контракт» — 50 000 ₽ / месяц, с 01.03 по 31.08
- **FR-03.3** Редактирование и удаление основного дохода. Не влияет на данные закрытых месяцев
- **FR-03.4** Учёт в бюджете текущего и всех будущих месяцев в пределах периода
- **FR-03.5** Приостановка основного дохода. Пауза действует до возобновления. Во время паузы доход не начисляется

#### Разовый (дополнительный) доход

- **FR-03.6** Добавление разового дохода для текущего месяца (название + сумма)
  > Пример: «Продажа велосипеда» — 15 000 ₽ — влияет только на текущий месяц
- **FR-03.7** Разовый доход влияет только на бюджет текущего месяца
- **FR-03.8** Использование разового дохода для ввода начального капитала при старте
  > Пример: «Текущий капитал» — 120 000 ₽ — пользователь создаёт при первом входе, если на счету уже есть средства

*Валидация: только положительные числа во всех денежных полях.*

---

### FR-04 · Управление ОЕП (Обязательные ежемесячные платежи)
*Регулярные обязательные расходы, повторяющиеся каждый месяц. Пользователь определяет критические статьи и ежемесячно подтверждает их оплату.*

- **FR-04.1** Создание бессрочного ОЕП (название + сумма)
  > Пример: «Аренда квартиры» — 45 000 ₽ / месяц
- **FR-04.2** Создание временного ОЕП (название, сумма, дата начала, дата окончания)
  > Пример: «Кредит на авто» — 18 000 ₽ / месяц, с 01.01.2024 по 31.12.2026
- **FR-04.3** Редактирование ОЕП с указанием, с какого месяца применять изменение. Не влияет на закрытые месяцы
- **FR-04.4** Разовое редактирование ОЕП — изменение суммы только в текущем месяце, не влияет на последующие
- **FR-04.5** Удаление / деактивация статьи ОЕП. Не влияет на закрытые месяцы. Прекращает будущие платежи
- **FR-04.6** Отображение в бюджете текущего и будущих месяцев
- **FR-04.7** Подтверждение (списание) каждого платежа пользователем. Статус: Запланирован → **Списан**
- **FR-04.8** Отмена платежа текущего месяца. Статус: Запланирован → **Отменён**. Списания с ВС не происходит

> **Примечание:** Изменение ОЕП не затрагивает закрытые месяцы. Перед закрытием система напоминает о необходимости проверки всех данных.

---

### FR-05 · Управление ПП (Плановые платежи)
*Периодические расходы с ежемесячным резервированием. Система разбивает крупный платёж на равные ежемесячные взносы, замораживая средства без списания с ВС.*

- **FR-05.1** Создание статьи ПП: название, целевая сумма, дата платежа, периодичность (в месяцах)
  > Пример: «ОСАГО» — 12 000 ₽, дата: 01.09, периодичность: 12 мес → ежемесячный взнос 1 000 ₽
- **FR-05.2** Автоматический расчёт ежемесячного взноса: Целевая сумма ÷ Периодичность
- **FR-05.3** Отображение прогресса: цель / накоплено / осталось / дата платежа
- **FR-05.4** При достижении цели: появляется кнопка **«Потратить»**. После нажатия — накопленная сумма списывается с ВС. До нажатия деньги остаются зарезервированными. Затем автоматически рассчитывается следующий цикл
- **FR-05.5** Редактирование с указанием, с какого месяца применять изменение. Не влияет на закрытые месяцы
- **FR-05.6** Разовое редактирование ПП — меняет взнос в текущем месяце, автоматически пересчитывает будущие
- **FR-05.7** Удаление / деактивация статьи ПП. Не влияет на закрытые месяцы

> **Примечание:** Редактирование ПП возможно даже если по нему уже были взносы — последующие платежи пересчитываются автоматически.

---

### FR-06 · Управление накоплениями
*Целевое откладывание средств с гибкими режимами. Доступно только при положительном Свободном остатке.*

> **Примечание:** Управление накоплениями становится доступным только если Свободный остаток > 0.

#### Режим 1 — «Как можно скорее»

- **FR-06.1** Ввод целевой суммы
- **FR-06.2** Ежемесячный платёж = Свободный остаток на момент создания накопления
  > Пример: СО = 20 000 ₽, цель = 100 000 ₽ → платёж 20 000 ₽ / мес, срок 5 мес
- **FR-06.3** Платёж пересчитывается только в меньшую сторону: если другие расходы уменьшили СО — платёж и срок корректируются. Если СО вырос — платёж не увеличивается автоматически (пользователь может создать новое накопление)

#### Режим 2 — «К определённой дате»

- **FR-06.4** Ввод целевой суммы и целевой даты
- **FR-06.5** Расчёт платежа = Целевая сумма ÷ Количество полных месяцев до даты
  > Пример: Цель 60 000 ₽ к 01.06, сейчас декабрь → 6 мес → платёж 10 000 ₽ / мес
- **FR-06.6** Валидация: платёж не должен превышать Свободный остаток

#### Режим 3 — «Без цели»

- **FR-06.7** Ввод фиксированной ежемесячной суммы
- **FR-06.8** Бессрочное накопление — без даты окончания
- **FR-06.9** Валидация: сумма не должна превышать Свободный остаток

#### Общее для всех режимов

- **FR-06.10** При достижении цели: появляется кнопка **«Потратить»**. Списание с ВС происходит только после ручного подтверждения. Статья накопления закрывается
- **FR-06.11** Отображение прогресса: накоплено / осталось / дата
- **FR-06.12** Отмена любого платежа. Система предлагает выбор: «Сместить срок» или «Увеличить последующие платежи» (если СО позволяет)

### FR-07 · Дашборд (главный экран)
*Единая точка контроля финансового состояния.*

**Структура:** таблица, где строки — статьи доходов и расходов, колонки — месяцы (3 закрытых + текущий + 12 будущих). Пересечение — запланированный или подтверждённый платёж.

- **FR-07.1** Ключевые показатели: Виртуальный счёт, Итого ОЕП, Итого ПП, Итого Накопления, Плановый свободный остаток, Реальный свободный остаток
- **FR-07.2** Список плановых доходов и расходов текущего месяца с итогами по категориям
- **FR-07.3** Статус каждого платежа: **Запланирован / Списан / Отменён**
- **FR-07.4** Статус каждого дохода: **Запланирован / Подтверждён**
- **FR-07.5** Детали по каждой статье ПП и Накоплений: цель, дата, накоплено, осталось
- **FR-07.6** Быстрое добавление разового дохода или незапланированного расхода
- **FR-07.7** Прогноз бюджетов на 12 месяцев вперёд: плановые показатели по каждому месяцу
- **FR-07.8** Просмотр закрытых месяцев: 3 месяца назад доступны как read-only
- **FR-07.9** Действие «Списать» напротив каждого платежа — подтверждение фактической оплаты
- **FR-07.10** Кнопка «Закрыть месяц» — финализация текущего периода
- **FR-07.11** Лог последних 50 действий пользователей — отдельный блок на дашборде

### FR-08 · Подтверждение операций
*Фиксация реальных финансовых событий. Все операции имеют три состояния: Запланирован / Списан / Отменён.*

#### Действия пользователя

- **FR-08.1** **«Списать»** — плановая сумма принята как фактическая. ВС уменьшается. Статус: Списан
- **FR-08.2** **«Изменить»** — ввод фактической суммы с автоматическим пересчётом. Статус: Списан на фактическую сумму
- **FR-08.3** **«Отменить»** — платёж не был совершён. Статус: Отменён. ВС не изменяется
- **FR-08.4** Пока месяц не закрыт — любое действие можно откатить и начать заново. После закрытия — недоступно

#### Логика отклонений для ОЕП

- **FR-08.5** Факт ≤ план (включая отмену): ВС уменьшается на фактическую сумму. Разница не переносится автоматически
- **FR-08.6** Факт > план: ВС уменьшается на фактическую сумму

#### Логика отклонений для ПП и Накоплений

- **FR-08.7** Факт < план: система предлагает — «Сместить дату цели» или «Увеличить последующие платежи» (если прогнозный СО позволяет в каждом оставшемся месяце)
- **FR-08.8** Факт > план: пользователь выбирает — «Ускорить цель» или «Уменьшить платёж»

#### Подтверждение доходов

- **FR-08.9** Доход подтверждается отдельным действием. После подтверждения ВС увеличивается. Статус: Подтверждён
- **FR-08.10** Неподтверждённый доход отображается в Плановом СО, но не влияет на ВС и Реальный СО

### FR-09 · Закрытие месяца
*Финальная фиксация периода. Переход к следующему месяцу.*

- **FR-09.1** Все плановые операции (доходы и расходы) должны иметь статус Списан/Подтверждён или Отменён. Наличие операций в статусе **Запланирован** блокирует закрытие
- **FR-09.2** Добавление незапланированных доходов и расходов — доступно до закрытия
- **FR-09.3** Свободный остаток после всех подтверждений автоматически переносится как дополнительный доход в следующий месяц. Пользователь видит накопленный эффект в будущих месяцах
- **FR-09.4** Опция **«Списать свободный остаток»**: означает, что деньги потрачены без детализации. ВС уменьшается на СО. В следующий месяц перенос не происходит
- **FR-09.5** Валидация: закрытие блокируется при отрицательном Свободном остатке
- **FR-09.6** Кнопка «Закрыть месяц» — финальное действие с предупреждением о необратимости
- **FR-09.7** Снимок закрытого месяца сохраняется в историю (read-only)
- **FR-09.8** Следующий месяц становится активным. Таблица сдвигается: старый -3 уходит, новый +12 добавляется

> **Примечание:** Закрытие месяца доступно всем участникам.

## 6. Бизнес-логика и алгоритмы расчётов

### 6.1 Основные принципы системы

1. Виртуальный счёт всегда соответствует реальному банковскому счёту — пользователь является связующим звеном
2. Резервирование не создаёт «денег из воздуха» — деньги либо свободны, либо зарезервированы, либо списаны
3. Свободный остаток = деньги, которые реально можно потратить прямо сейчас
4. Месяц не может быть закрыт с отрицательным СО или с операциями в статусе Запланирован
5. Закрытые месяцы неизменны — история защищена от редактирования

### 6.2 Формулы расчёта

```
// ── ДОХОДЫ ──────────────────────────────────────────────────────────
Доход плановый  = Σ(Основной доход всех участников за период)
                + Σ(Разовые доходы текущего месяца)

Доход реальный  = Σ(Только подтверждённые доходы)

// ── РАСХОДЫ (план) ───────────────────────────────────────────────────
ОЕП план        = Σ(Все активные статьи ОЕП в текущем месяце)
ПП план         = Σ(Ежемесячные взносы по активным статьям ПП)
Накопления план = Σ(Ежемесячные взносы по активным накоплениям)

// ── ВИРТУАЛЬНЫЙ СЧЁТ ─────────────────────────────────────────────────
ВС = Начальный баланс (перенесён из предыдущего месяца)
   + Σ(Все подтверждённые доходы месяца)
   − Σ(Все подтверждённые ОЕП, ПП и Накопления — «Списать»)
   − Σ(Свободный остаток, если выбрано «Списать СО» при закрытии)

// ── ЗАРЕЗЕРВИРОВАННЫЕ СРЕДСТВА ───────────────────────────────────────
Резерв = Σ(Подтверждённые, но ещё не списанные взносы по ПП и Накоплениям)

// ── СВОБОДНЫЙ ОСТАТОК ────────────────────────────────────────────────
СО реальный  = ВС − Резерв
СО плановый  = (Доход плановый − ОЕП план − ПП план − Накопления план)
             + Перенесённый СО из предыдущего месяца

// ── ЗАКРЫТИЕ МЕСЯЦА ──────────────────────────────────────────────────
Если «Перенести СО»:  следующий месяц += СО реальный текущего месяца
Если «Списать СО»:    ВС -= СО реальный;  в следующий месяц не переносится
```

### 6.3 Обработка отклонений от плана

#### Для ОЕП
- Факт ≤ план (включая отмену): ВС уменьшается на фактическую сумму. Разница не переносится автоматически
- Факт > план: ВС уменьшается на фактическую сумму

#### Для ПП и Накоплений
- Факт < план: пользователь выбирает — «Сместить дату» или «Увеличить последующие платежи»
  - Новая дата: `Текущая дата + ⌈Оставшаяся сумма ÷ Текущий платёж⌉`
  - Увеличение платежа: доступно, если прогнозный СО каждого оставшегося месяца позволяет
- Факт > план: пользователь выбирает — «Ускорить цель» или «Уменьшить платёж»

#### Для накопления «Как можно скорее»
- Платёж пересчитывается только в меньшую сторону при уменьшении СО
- При росте СО — платёж не увеличивается автоматически; пользователь создаёт новое накопление при желании

- Поддерживаемые браузеры: современные Chromium-based (Chrome, Edge, Brave и аналоги)

- Язык интерфейса: русский
